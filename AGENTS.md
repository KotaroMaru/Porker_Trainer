# AGENTS.md

正典。Claude Code・Codex 両方のAIエージェントが従う。

## プロジェクト概要
ポーカーGTO(Game Theory Optimal)ポストフロップトレーナー。React 19 + Vite + TypeScript製PWA。
フロップは事前計算(Rust)、ターン/リバーはブラウザ内TS CFRソルバーでライブ解析する。

## スタック・構成
- React 19 / Vite 8 / TypeScript 6 (strict) / Zustand 5 / vite-plugin-pwa
- テスト: Vitest 4 + Testing Library + jsdom。Lint: ESLint 10 flat config。Prettier未導入
- パッケージマネージャ: npm(`package-lock.json`)。Nodeバージョン未固定(`.nvmrc`等なし)
- `src/gto/` — GTOトレーナー本体(solver/tree/worker/loader/trainer/explain/data/store.ts)
- `src/engine/`, `src/analysis/` — ポーカー基礎ロジック・レンジ処理
- `tools/solver/` — Rustワークスペース(AGPL-3.0-or-laterのpostflop-solverに依存、ビルド時専用・
  アプリ非リンク)。詳細は `tools/solver/README.md`
- `.github/workflows/gto-batch.yml` — ソルバーバッチ生成CI(唯一のワークフロー)

## コマンド(実行確認済み)
- 型チェック: `npx tsc -b` / テスト全体: `npm test` / 単一: `npx vitest run <path>`
- Lint: `npm run lint`(既存の技術的負債あり=52 errors/11 warnings(**AGENTS.md作成時の基準値**、
  恒久的な正解値ではない)。**変更ファイル・変更箇所に起因する新規エラーを追加しない**。
  全体件数だけでなく今回の変更起因かを確認し、タスク外の既存Lint修正へ無制限に範囲を広げない)
- ビルド: `npm run build`(型チェック→vite build→PWAプリキャッシュ自動検査)

## コーディング規約
- import: 相対パスのみ(パスエイリアスなし)。型は `import type {...}` で明示
- 命名: コンポーネント=PascalCase.tsx、ロジック=camelCase.ts。export は名前付きのみ
- コメントは「なぜ」を書く(日本語主体)。既存パターンを踏襲し、新しい抽象化は必要になってから

## ★GTOドメイン最重要ルール(最優先事項)
`src/gto/solver/`・`src/analysis/`・`src/engine/`のエクイティ計算・レンジ vs レンジ処理・
GTO戦略/ソルバーは**「動くこと」より「数値的に正しいこと」が最優先**。UIが動いても
計算が誤っていれば不合格。**配置場所にかかわらず**、確率・エクイティ・EV・レンジ・
コンボ・ゲーム木・CFR更新・戦略頻度を扱うコードにはこのルールを適用する。

この領域を変更したら、以下を**必ず**テストで検証すること:
- 既知の解析解との比較。例: 現在の実装と同じ利得定義(アンティ1・ベット1)・P0(先手)視点での
  Kuhnポーカーのゲーム値 -1/18(`src/gto/solver/cfr.test.ts`で確認済み)。**期待値の符号と
  プレイヤー視点を必ずテスト内に明記する**
- 頻度の**正規化**(合計1になるか)、レンジの**コンボ数**、**カード重複**(ブロッキング)の扱い
- **ブロッカー**効果、手の**対称性**(スート対称等)、**tie率**(引き分け)の扱い
- **浮動小数点の許容誤差**を明示した比較(厳密等価で比較しない)
- Monte Carloを使う場合: **seed固定・試行回数固定・許容誤差または信頼区間を明記**
- 小規模ケースは**全列挙**(exhaustive enumeration)の結果と突き合わせる
- Rust版との交差検証(`tools/solver/crossvalidation/`)
- CFR・戦略更新を変更する場合、**利用可能な範囲で**exploitability・NashConv・regret・
  既知ゲームのゲーム値など実装から独立した収束指標を確認する。反復回数だけを収束の
  根拠にしない。収束閾値を変更する場合は速度と精度への影響を記録する

採点ロジック(頻度ベース、過去に承認済み)やCFRの収束判定は軽々しく変更しない。

## Claude CodeからCodexプラグインを使う場合
- 小さな差分(1〜2ファイル程度): `/codex:review` を**foregroundで1回**
- 複数ファイルの差分・長時間になりそうなレビュー: `--background` を使う
- Codexへ渡す内容は短いタスクパケット形式にする: **目的 / 対象ファイル / 変更可能範囲 /
  禁止事項 / 受け入れ条件 / 検証コマンド**
- review gateは有効化しない。レビューは必要なタイミングで手動実行する

## Claude・Codex協調ルール

**役割分担**: Claudeは設計・タスク分割・高リスク判断に専念する。調査・実装・テスト・軽微な
修正はCodexが1スレッド内で完結させる。

**委譲時に一度に渡す**: 目的・対象範囲・変更禁止範囲・受け入れ条件・検証コマンド
(上記のタスクパケット形式)。

**コンテキストの節約**: Codexが既に調査済みのファイルをClaudeが無条件に再調査しない。
完全なdiff・コード全文・長いログをClaudeのコンテキストへ取り込まない。

**Codex→Claude報告は以下のみに限定**: 変更概要 / 変更ファイル / 検証結果 /
数値検証方法 / 未解決事項 / Claudeの判断が必要な事項。

**Claudeへ戻す主な条件**:
仕様の曖昧さ / 公開API・保存形式の変更 / 要求とテストの矛盾 /
GTO既知値との不一致 / 同一問題への2回の失敗 / 対象範囲の拡大 / セキュリティ判断が必要 /
ライセンス上の判断が必要 / データ損失の可能性 / 重大な性能劣化の可能性 /
タスクパケットの前提と実際のコードが矛盾 / 既存の未コミット変更と衝突 /
不可逆または高リスクな判断が必要 / 推測による実装を続ける必要がある。
上記に列挙されていない場合でも、**不可逆・高リスクな判断が必要なときは作業を止めて
エスカレーションする**。

**HANDOFF.md**: 履歴を蓄積するファイルではなく、**現在の引き継ぎ状態だけ**を保持する。
未完了の非軽微タスクを中断するとき・別エージェントへ切り替えるとき・Claudeの利用制限が
近いとき・ユーザーから明示されたときに更新する。内容は目的・完了事項・残タスク・
変更ファイル・重要判断・検証結果・禁止範囲のみ(完了済みタスクの詳細は蓄積しない)。

**この協調ルール自体の変更**: 実行回数・往復回数・再作業・テスト結果の記録から
改善案を提案してよい。ただしAGENTS.md・品質基準・モデル方針を自動変更せず、
必ずユーザー承認を得る。

## 作業開始・終了時
- 作業開始前に `git status --short` を確認する
- ユーザーや他エージェントの既存変更を自分の変更として扱わない
- 無関係な既存変更を修正・整形・削除・revertしない
- 変更はタスクに必要な最小範囲に限定する
- 完了時は変更ファイル・実行した検証・失敗した検証・未解決事項を報告する
- 実行できなかったテストやコマンドを、実行済みと報告しない

## やってはいけないこと
- テストを書かずに `src/gto/solver/`・採点ロジック・エクイティ計算を変更しない
- 既存の頻度ベース採点方式を無断で変更しない
- `tools/solver/vendor/postflop-solver/` を改変しない(AGPLのサードパーティ製、rev固定)
- `eslint-disable` の濫用で既存エラーを隠さない
- path aliasやPrettier導入などツール構成を独断で変更しない
- 未承認のまま `git push` や破壊的なgit操作(reset --hard/clean/force push等)を行わない
- `.env`・APIキー・アクセストークン・認証情報を表示・コミット・ログへ記録しない
- 設定例では実値を使わず、`.env.example`とダミー値を使用する
